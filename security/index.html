What's actually true about our security

We'd rather tell you exactly what's implemented than make a vague claim that sounds reassuring but doesn't hold up. Here's the real picture, tool by tool.

🔐 Scrypt password hashing 🔍 Zero secrets found in any repo (audited) 📖 Open-source core (MIT) 🚫 No data sold, ever

The honest headline

We do not hold SOC 2, ISO 27001, or any third-party security certification. We're a small, independent toolmaker, not an enterprise vendor — and we'd rather say that plainly than imply otherwise with vague "bank-level security" language. What follows is a specific, verifiable account of what each tool actually does.

What's true for every Oli tool

  • Passwords are never stored in plain text. Every product that has user accounts (OliOps/OliCRM, OliCompute, OliFlow, Oli-Locator's backend, project-2/OliMind) hashes passwords with scrypt (a memory-hard, industry-standard algorithm) or bcrypt, each with a unique per-user salt.
  • Session tokens use timing-safe comparison. Login sessions are verified using constant-time comparison (crypto.timingSafeEqual), which protects against timing-attack side channels.
  • We manually audited every repo for leaked secrets — API keys, private keys, hardcoded credentials — before any tool was made available. None were found in the current codebases. (One historical issue was found and fixed — see "Past incident, disclosed" below. We're including it because hiding it would undercut everything else on this page.)
  • Core code is open-source (MIT licensed). You — or a security professional you hire — can read exactly how each tool handles your data, rather than trusting a black box.
  • We do not sell customer data. Not to advertisers, not to data brokers, not to anyone. See the Privacy Policy for exactly what we do collect and why.

Self-hosted tools: what "self-hosted" means for your security

OliOps Suite, OliCommerce Stack, OliFlow Engine, and OliConnect's core engines are self-hosted — you run them on your own server or hosting provider, not ours. This has real security implications in both directions, and we want you to understand both:

What this meansUpsideWhat's on you
Your data lives on infrastructure you chooseWe never have access to your customer/business data — there's no central Oli database to breachYou're responsible for keeping that infrastructure patched and secured
You control HTTPS/TLS setupYou can use your own certificate provider (Let's Encrypt is free) and your own hosting security postureThe app itself doesn't force HTTPS — you need to configure it (see setup guide below)
You control backupsNo third party has copies of your backupsIf you don't back up your own data directory, we can't recover it for you

Recommended hardening checklist (5 minutes, once)

  • Put your instance behind HTTPS — most hosts (Render, Railway, Vercel) provide this automatically; if self-hosting on a VPS, use Let's Encrypt (free)
  • Set a strong, unique AUTH_SECRET / signing secret environment variable — never leave it as a default/example value in production
  • Back up your data directory on a schedule (a simple daily cron + off-site copy is enough for most small businesses)
  • Keep Node.js and your OS patched — these tools use zero or minimal npm dependencies specifically to shrink this burden, but the runtime itself still needs updates

OliOps Suite & Oli-Locator's hosted option

If you use the optional managed-hosting add-on (rather than self-hosting) for OliOps Suite, or if you use Oli-Locator (which is hosted-only, login-based SaaS), here's what applies specifically:

  • We manage TLS/HTTPS termination for you on hosted instances
  • Data is stored on the hosting provider's infrastructure we've configured (currently: free/low-cost tiers of mainstream providers — see Privacy Policy for the current list)
  • We do not currently run a formal penetration test program — this is realistic for our current size, and we'll update this page the day that changes
  • We do not offer a bug bounty program yet — if you find a security issue, please use responsible disclosure (see below) and we'll credit you here once we have a formal program

OliConnect (browser extension) specifically

Your social media login credentials never touch our servers. OliConnect operates entirely within your own authenticated browser session — it does not ask for, store, or transmit your passwords anywhere. Post content it reads/recycles stays local to your browser and the platforms you're already logged into.

Past incident, disclosed

What happened: during development, an earlier version of the Oli-Locator (lead-gen) codebase had a real owner-account password hardcoded in plain text inside client-side JavaScript, plus a matching password hash in the server code.

What we did: removed the hardcoded credential entirely, moved account seeding to environment variables never committed to source, and treated the exposed password as permanently compromised (rotated, not reused). This is documented in the repository's commit history for anyone who wants to verify it independently.

Why we're telling you: a security page that only lists good news isn't credible. If we find something wrong, we'll say so here — that's the only way "trust" on this page means anything.
